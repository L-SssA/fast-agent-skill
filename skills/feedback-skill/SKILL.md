# Feedback Skill - 技能反馈提取助手

本技能用于系统化地从聊天上下文中提取用户对技能的纠正和反馈，识别技能不完善之处，并将这些洞察转化为可操作的规范化改进建议，最终反馈到对应的技能包中。

## 前置条件

使用本技能前，确保：

- 存在包含技能使用过程的完整聊天上下文
- 用户在任务执行过程中有过纠正行为或明确反馈
- 能够识别出涉及的具体技能名称
- 目标技能包已存在且可以正常访问

**如果尚未创建技能**，请先使用 create-skill 技能搭建基础框架。

## 使用流程

**第零步: 确认反馈目标 (必需)**

在开始分析前,**必须**明确以下信息:

1. **哪个技能需要改进?**
   - 从对话中识别被纠正的技能名称
   - 如果不确定,向用户确认:"请问您希望改进哪个技能?"
   - **重要**: 区分"创建的技能"和"正在使用的技能"
     - 例如: 创建 tech-resume-optimizer 时的纠正 → 应反馈到 create-skill
     - 例如: 使用 csv-analyzer 时的纠正 → 应反馈到 csv-analyzer

2. **纠正是关于什么的?**
   - 是关于技能本身的功能缺陷?
   - 还是关于使用该技能时的操作问题?
   - 或者是技能创建过程中的流程问题?

**判断规则**:

- ✅ 如果用户在"创建技能"过程中纠正你 → 反馈目标是 **create-skill**
- ✅ 如果用户在使用某个具体技能时纠正你 → 反馈目标是 **该技能**
- ❌ 不要将创建过程的反馈应用到被创建的技能上

**示例**:

```
场景: 用户让你用 create-skill 创建 tech-resume-optimizer
纠正: "你应该运行验证脚本"
分析: 这是 create-skill 工作流程的缺失,不是 tech-resume-optimizer 的问题
反馈目标: create-skill ✓
```

1. **分析聊天上下文**: 扫描对话历史，识别通过技能完成的任务
2. **检测纠正行为**: 查找用户纠正 agent 的交互片段
3. **归因分析**: 判断纠正是否由技能不完善导致
4. **提取改进点**: 从纠正中提取可规范化的具体问题和解决方案
5. **生成分类报告**: 按问题类型组织改进建议
6. **交互式确认**: 向用户展示提取结果，获得采纳确认

**确认方式**:

**推荐**: 使用 IDE 的 `AskUserQuestion` 工具进行结构化提问

当有多个改进点需要确认时,**优先使用** `AskUserQuestion` 工具,它会渲染一个结构化的交互组件,提供清晰的选项。

**工具调用格式**:

```json
{
  "questions": [
    {
      "question": "您希望应用哪些改进点到 feedback-skill?",
      "header": "应用范围",
      "multiSelect": false,
      "options": [
        {
          "label": "全部应用 (推荐)",
          "description": "应用所有 P1/P2/P3 改进点,全面提升技能质量"
        },
        {
          "label": "仅应用 P1 改进",
          "description": "只修复最关键的反馈目标确认问题"
        },
        {
          "label": "自定义选择",
          "description": "我将逐个确认每个改进点"
        }
      ]
    }
  ]
}
```

**核心参数说明**:

| 字段                | 类型    | 说明                                     |
| ------------------- | ------- | ---------------------------------------- |
| question            | string  | 完整的问题文本,应以问号结尾              |
| header              | string  | 显示为标签的简短标题,最多 12 字符        |
| multiSelect         | boolean | 是否允许多选,默认 false                  |
| options             | array   | 2-4 个选项,每项包含 label 和 description |
| options.label       | string  | 选项的显示文本                           |
| options.description | string  | 选项的详细描述                           |

**优势**:

- ✅ 渲染为结构化 UI 组件,比纯文本更清晰直观
- ✅ 提供预设选项,减少用户的认知负担
- ✅ 支持多选/单选,灵活适应用户需求
- ✅ 用户始终可以选择"Other"输入自定义文本
- ✅ 在关键决策点引入人类判断,减少猜测和返工

**适用场景**:

- 需求模糊,存在多种合理解读
- 存在多个同等合理的技术方案
- 即将执行高风险或不可逆操作(如修改技能文件)
- 需要用户做出偏好决策

**备选**: 如果改进点较少(1-2个),可以直接展示并询问:

```markdown
**请确认:**

1. 是否同意这些改进点?
2. 希望现在应用还是稍后处理?
```

**注意事项**:

- 对于复杂的多项改进,**必须**使用 `AskUserQuestion` 工具
- 每个选项的描述要清晰说明后果和影响
- 提供"推荐"选项帮助用户快速决策
- 如果用户选择"自定义",再逐个确认每个改进点
- **不要**假设用户的选择,必须等待用户明确响应

7. **应用反馈**: 将确认的改进点整合到对应技能中
8. **验证改进**: 建议用户使用 optimize-skill 测试验证

## 关键能力摘要

### 上下文分析

- 扫描完整对话历史，识别技能调用点
- 追踪任务执行流程和 agent 响应
- 定位用户纠正、质疑或不满的表达
- 提取关键的交互转折点和失败案例

### 纠正检测

- 识别显式纠正："不对"、"应该是"、"你理解错了"等
- 识别隐式纠正：用户重新描述需求、提供修正示例
- 识别不满意信号：重复询问、表达困惑、要求重试
- 区分技能缺陷 vs 用户误解 vs 外部因素

### 归因分析

- 判断纠正是否源于技能描述不清
- 判断纠正是否源于指令缺失或不准确
- 判断纠正是否源于缺少边界情况说明
- 判断纠正是否源于示例不足或误导
- 排除非技能因素（用户输入模糊、外部环境变化等）

### 反馈提取

- 提取具体的问题描述和改进方向
- 分类问题类型：描述优化、指令补充、边界情况、示例增强等
- 生成结构化的改进建议（问题 → 原因 → 解决方案）
- 保持建议的可操作性和具体性

### 智能整合

- 决策改进点应添加到 SKILL.md 还是 references/
- 选择合适的位置插入新内容
- 维护渐进式披露原则
- 避免内容重复和冲突
- **保持行文一致性**:
  - 检查原文档的结构风格(列表/章节/段落)
  - 新内容应与现有风格保持一致
  - 避免将简洁的列表项扩展为冗长章节
  - 如需详细说明,优先使用子列表而非独立章节
  - 修改后通读全文,确保逻辑流畅

**行文一致性检查清单**:

- [ ] 新内容的层级结构与原文一致
- [ ] 没有破坏原有的列表/章节组织
- [ ] 长度适中,符合渐进式披露原则
- [ ] 语言风格与原文统一

## 常见应用场景

### 场景 1：用户纠正了技能的理解偏差

**问题**：用户使用技能时，agent 误解了任务意图，用户进行了纠正  
**解决方案**：

1. 分析对话，定位纠正点
2. 加载 references/context-analysis.md 进行归因分析
3. 判断是否因技能描述不清导致误解
4. 提取改进建议：优化 description 或添加澄清说明
5. 用户确认后，加载 references/feedback-application.md 应用到技能

### 场景 2：技能遗漏了重要边界情况

**问题**：技能在特定边界条件下表现不佳，用户指出了这个问题  
**解决方案**：

1. 识别边界情况的描述和处理方式
2. 加载 references/feedback-extraction.md 提取规范化建议
3. 创建或更新 references/ 中的边界情况文档
4. 在 SKILL.md 中添加对边界的引用

### 场景 3：技能缺少关键步骤或指令

**问题**：agent 执行任务时遗漏了重要步骤，用户补充了指令  
**解决方案**：

1. 提取用户补充的关键步骤
2. 判断应添加到 SKILL.md 的工作流还是 references/ 的详细指南
3. 更新相应的指令部分
4. 确保步骤顺序和逻辑清晰

### 场景 4：技能示例不够充分或存在误导

**问题**：技能提供的示例不足以覆盖实际场景，或有误导性  
**解决方案**：

1. 收集用户提供的正确示例或指出示例问题
2. 创建新的示例文档或更新现有示例
3. 确保示例覆盖典型场景和边界情况
4. 在 SKILL.md 中更新示例引用

## 工作流程示例

```
用户: "刚才用 csv-analyzer 技能处理数据时，我纠正了它三次，感觉这个技能有问题"

你:
1. 确认要分析的技能是 "csv-analyzer"
2. 扫描最近的聊天上下文，找到使用该技能的对话
3. 加载 references/context-analysis.md 进行分析
4. 识别出三次纠正：
   - 纠正 1: agent 错误理解了日期格式 → 技能缺少格式说明
   - 纠正 2: agent 遗漏了空值处理 → 技能未提及边界情况
   - 纠正 3: agent 输出格式不符合要求 → 技能示例不清晰
5. 加载 references/feedback-extraction.md 提取改进建议
6. 生成反馈报告：
   - 【描述优化】在 description 中添加日期格式处理能力说明
   - 【边界情况】创建 references/data-validation.md 说明空值处理
   - 【示例增强】更新 output-format 示例，展示正确的输出格式
7. 向用户展示报告，请求确认
8. 用户确认所有改进点
9. 加载 references/feedback-application.md 应用改进
10. 执行应用：
    - 优化 SKILL.md 的 description
    - 创建 references/data-validation.md
    - 更新示例章节
11. 生成应用报告
12. 建议使用 optimize-skill 进行测试验证
```

## 与相关技能的关系

- **create-skill**：从零创建新技能的基础框架
- **optimize-skill**：系统化优化技能质量、描述准确性和性能
- **enhance-skill**：从外部资料中提取信息并增强技能内容
- **feedback-skill**：从实际使用中提取用户反馈，持续改进技能

**典型工作流**：

1. 使用 create-skill 创建基础框架
2. 在实际使用中收集用户反馈
3. 使用 feedback-skill 从对话中提取改进点
4. 应用反馈到技能中
5. 使用 optimize-skill 进行系统化测试验证
6. 根据需要重复步骤 2-5 持续迭代

四个技能协同工作，形成技能创建 → 使用 → 反馈 → 优化 → 验证的完整闭环。

## 重要提醒

- **始终进行交互式确认**：不要自动应用未经用户确认的改进
- **准确归因**：仔细区分技能缺陷和其他因素导致的纠正
- **保持具体性**：提取的改进点必须具体、可操作，避免泛泛而谈
- **遵循渐进式披露**：详细内容放入 references/，保持 SKILL.md 简洁
- **注意内容去重**：避免添加与现有内容重复的改进
- **维护引用关系**：新增 reference 文件时，确保在 SKILL.md 中正确引用
- **关注可测试性**：提取的改进点应该能够通过 optimize-skill 进行验证
- **遵循精简原则**:
  - 优先提供必要信息,避免过度详细
  - 对于通用知识(如Python环境),简要提及即可
  - 只在必要时才提供多种方案,默认推荐一种
  - 问自己:"这个细节真的需要吗?"如果不确定,选择不添加

记住：你的目标是帮助用户**从实际使用中系统化地提取反馈**，将用户的纠正转化为技能的持续改进，形成良性循环。
