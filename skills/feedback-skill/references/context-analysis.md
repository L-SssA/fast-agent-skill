# Context Analysis - 上下文分析方法论

本文档提供系统化的聊天上下文分析方法，用于识别技能使用过程中的问题和用户纠正行为。

## 对话历史扫描策略

### 1. 定位技能调用点

**目标**：找到所有涉及技能使用的对话片段

**方法**：

- 搜索技能名称或相关关键词的出现位置
- 识别 agent 加载技能的标记（如 "Loading skill..."）
- 追踪技能激活后的任务执行流程
- 记录每次技能调用的输入和输出

**示例模式**：

```
用户: "帮我分析这个 CSV 文件"
Agent: [加载 csv-analyzer 技能]
Agent: [执行分析...]
用户: "不对，日期格式应该是 YYYY-MM-DD"  ← 纠正点
```

### 2. 追踪任务执行流

**关键要素**：

- 用户的初始请求
- Agent 的理解和计划
- 执行的步骤和工具调用
- 中间结果和最终输出
- 用户的反馈和纠正

**分析方法**：

1. 按时间顺序梳理对话
2. 标记每个关键转折点
3. 识别偏离预期的地方
4. 定位用户介入的时机

## 纠正行为识别模式

### 显式纠正信号

**直接否定**：

- "不对"、"错了"、"不正确"
- "这不是我想要的"
- "你理解错了"
- "重新来"

**修正指令**：

- "应该是..."
- "正确的做法是..."
- "实际上需要..."
- "换个方式..."

**质疑表达**：

- "为什么这样做？"
- "这样对吗？"
- "确定吗？"
- "有没有考虑...？"

### 隐式纠正信号

**重新描述需求**：

- 用户再次解释任务，强调之前遗漏的点
- 提供更详细的说明或约束条件
- 澄清模糊的表述

**提供修正示例**：

- 用户给出期望的输出样例
- 展示正确的处理方式
- 对比错误和正确的结果

**重复询问**：

- 对同一问题多次提问
- 要求确认某个细节
- 要求重试某个步骤

**表达困惑或不满**：

- "这好像不太对"
- "感觉有问题"
- "能不能换种方法"
- "我还是不太明白"

### 不满意程度评估

**轻度不满意**：

- 轻微调整建议
- 补充说明
- 询问替代方案

**中度不满意**：

- 明确指出错误
- 要求重新执行
- 提供详细纠正

**重度不满意**：

- 多次纠正仍不正确
- 表达强烈不满
- 放弃使用当前方法

## 归因分析框架

### 判断纠正原因的分类体系

#### 1. 技能描述不清 (Description Issue)

**特征**：

- Agent 误解了技能的适用范围
- 在不应使用该技能的场景下激活
- 对用户意图的理解偏差

**判断标准**：

- 检查 description 是否包含相关关键词
- 评估描述的边界是否清晰
- 分析是否有歧义表述

**示例**：

```
用户: "帮我整理这些笔记"
Agent: [激活 note-organizer 技能，但用户想要的是代码重构]
→ 原因：description 中缺少"仅适用于文本笔记，不适用于代码"的说明
```

#### 2. 指令缺失或不准确 (Missing/Inaccurate Instructions)

**特征**：

- Agent 遗漏了关键步骤
- 执行顺序错误
- 使用了错误的方法或工具

**判断标准**：

- 检查 workflow 是否完整
- 验证步骤是否清晰明确
- 评估是否有默认方案

**示例**：

```
用户: "分析这个数据集"
Agent: [直接开始分析，未检查数据质量]
用户: "应该先处理缺失值"
→ 原因：技能指令中缺少数据预处理步骤
```

#### 3. 边界情况未覆盖 (Missing Edge Cases)

**特征**：

- 在特殊输入下表现异常
- 未处理异常情况
- 对边界条件的假设错误

**判断标准**：

- 识别失败的具体场景
- 检查是否有 Gotchas 章节
- 评估边界情况的覆盖度

**示例**：

```
用户: "处理这个空文件"
Agent: [报错或产生无意义输出]
→ 原因：技能未说明如何处理空输入或异常情况
```

#### 4. 示例不足或误导 (Insufficient/Misleading Examples)

**特征**：

- Agent 模仿了错误的示例模式
- 示例未覆盖当前场景
- 示例与实际情况不符

**判断标准**：

- 检查示例的代表性
- 评估示例的完整性
- 验证示例的准确性

**示例**：

```
用户: "生成 JSON 输出"
Agent: [生成的 JSON 格式不符合预期]
用户: "应该是这样的格式：{...}"
→ 原因：技能中的示例格式不完整或有误
```

#### 5. 非技能因素 (Non-Skill Factors)

**排除情况**：

- 用户输入本身模糊或不完整
- 外部环境变化（API 变更、依赖更新等）
- Agent 的一般性理解错误（与技能无关）
- 用户改变了需求（而非纠正错误）

**判断方法**：

- 如果更换技能仍会出现同样问题 → 非技能因素
- 如果问题是普遍性的而非特定于该技能 → 非技能因素
- 如果用户最初的需求就不清晰 → 非技能因素

### 归因分析决策树

```
用户纠正发生
    ↓
是否与技能执行直接相关？
    ├─ 否 → 标记为"非技能因素"，跳过
    └─ 是 ↓
         是否是描述导致的误解？
            ├─ 是 → 标记为"描述问题"
            └─ 否 ↓
                 是否是指令缺失或错误？
                    ├─ 是 → 标记为"指令问题"
                    └─ 否 ↓
                         是否是边界情况？
                            ├─ 是 → 标记为"边界情况"
                            └─ 否 ↓
                                 是否是示例问题？
                                    ├─ 是 → 标记为"示例问题"
                                    └─ 否 → 标记为"其他/需人工判断"
```

## 问题分类体系

### 按严重程度分类

**P0 - 严重缺陷**：

- 导致技能完全无法使用
- 产生错误或有害的结果
- 多次纠正仍无法解决

**P1 - 重要问题**：

- 显著影响使用体验
- 需要用户频繁纠正
- 限制了技能的适用范围

**P2 - 一般改进**：

- 提升使用便利性
- 优化输出质量
- 增强鲁棒性

**P3 - 细微优化**：

- 文字表述改进
- 格式调整
- 小的用户体验提升

### 按问题类型分类

| 类型         | 子类型     | 典型表现               |
| ------------ | ---------- | ---------------------- |
| **描述问题** | 范围不清   | 技能在不相关场景被激活 |
|              | 关键词缺失 | 相关查询未触发技能     |
|              | 歧义表述   | Agent 误解技能用途     |
| **指令问题** | 步骤缺失   | 遗漏关键操作           |
|              | 顺序错误   | 执行流程不合理         |
|              | 方法不当   | 使用了低效或错误的方法 |
| **边界情况** | 异常输入   | 空值、错误格式等未处理 |
|              | 极端场景   | 大数据量、特殊情况等   |
|              | 集成问题   | 与其他工具/技能的交互  |
| **示例问题** | 示例不足   | 未覆盖典型场景         |
|              | 示例错误   | 示例本身有误           |
|              | 格式不清   | 输出格式不明确         |
| **其他**     | 性能问题   | 执行效率低             |
|              | 可用性问题 | 使用不便               |
|              | 文档问题   | 说明不清楚             |

## 分析工作流

### Step 1: 收集对话数据

```markdown
1. 确定分析的时间范围
2. 提取完整的对话历史
3. 标记技能相关的片段
4. 建立时间线
```

### Step 2: 识别纠正点

```markdown
1. 扫描显式纠正信号
2. 检测隐式纠正信号
3. 评估不满意程度
4. 记录每个纠正点的上下文
```

### Step 3: 归因分析

```markdown
1. 对每个纠正点进行原因分析
2. 使用决策树进行分类
3. 排除非技能因素
4. 确定问题类型和严重程度
```

### Step 4: 生成分析报告

```markdown
1. 汇总所有纠正点
2. 按类型和严重程度分组
3. 提取关键洞察
4. 形成初步的改进建议方向
```

## 实用技巧

### 提高识别准确率

1. **关注语气变化**：用户从配合到质疑的转变往往是纠正信号
2. **注意重复模式**：同一问题的多次出现表明系统性缺陷
3. **分析纠正深度**：浅层纠正可能是小问题，深层纠正可能是设计缺陷
4. **追踪解决路径**：观察用户如何引导 agent 回到正确轨道

### 避免误判

1. **区分纠正和改进建议**：用户说"可以试试..."不一定是纠正
2. **考虑用户学习曲线**：新用户可能需要更多指导，不一定是技能问题
3. **评估上下文完整性**：确保理解了完整的对话背景
4. **多方验证**：如果有多个类似案例，更有说服力

### 量化分析

**指标建议**：

- 纠正频率：单位时间内纠正次数
- 纠正深度：平均需要几轮对话才能解决
- 成功率：纠正后是否真正解决问题
- 用户满意度：从语气和用词推断

**计算公式**：

```
技能缺陷指数 = (纠正次数 × 严重程度权重) / 总使用次数

严重程度权重：
- P0: 4
- P1: 3
- P2: 2
- P3: 1
```

## 总结

上下文分析是反馈提取的基础，关键在于：

1. **全面扫描**：不遗漏任何纠正信号
2. **准确归因**：正确判断问题根源
3. **系统分类**：便于后续的规范化处理
4. **量化评估**：为优先级排序提供依据

通过系统化的分析，可以将零散的用户纠正转化为结构化的改进洞察，为技能优化提供可靠的数据支持。
