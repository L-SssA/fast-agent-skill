# Feedback Extraction - 反馈提取和规范化指南

本文档提供从用户纠正中提取改进点并将其规范化的系统方法，确保反馈具有可操作性和高质量。

## 改进点提取方法

### 从纠正到改进的转化流程

```
用户纠正 → 理解问题本质 → 抽象为通用规则 → 形成具体建议 → 规范化表述
```

### Step 1: 理解问题本质

**目标**：透过表面现象，找到根本原因

**方法**：

1. **重述问题**：用自己的话重新描述用户指出的问题
2. **追问为什么**：连续问"为什么会这样"直到找到根因
3. **对比期望与实际**：明确用户期望的行为和实际行为的差异
4. **识别模式**：这是孤立问题还是系统性缺陷

**示例**：

```
用户纠正: "日期格式应该是 YYYY-MM-DD，不是 MM/DD/YYYY"

理解过程：
- 表面问题：日期格式错误
- 深层原因：技能未指定日期格式标准
- 根本原因：缺少输入/输出格式的明确说明
- 模式判断：可能还有其他格式相关的未说明项
```

### Step 2: 抽象为通用规则

**目标**：将具体问题提升为可复用的规范

**抽象层次**：

- **太具体**：仅适用于当前案例，价值有限
- **太抽象**：过于宽泛，难以执行
- **恰到好处**：既解决当前问题，又有普适性

**示例**：

```
具体问题: "处理 CSV 时应该先检查空值"

太具体的抽象: "在这个特定的 CSV 文件中要检查空值"
  → 仅适用于这一个文件

太抽象的抽象: "总是进行数据验证"
  → 不够具体，不知道如何执行

恰到好处的抽象: "在处理任何数据集前，应先进行数据质量检查，包括：
  - 检测和处理缺失值
  - 验证数据类型
  - 检查异常值"
  → 既解决了当前问题，又提供了通用指导
```

### Step 3: 形成具体建议

**建议的结构**：

```markdown
**问题**: [清晰描述当前的问题或缺失]

**影响**: [说明这个问题导致的后果]

**解决方案**: [具体的改进措施]

**示例**: [如果适用，提供示例]

**位置**: [建议添加到技能的哪个部分]
```

**质量标准**：

- ✅ 具体可执行
- ✅ 有明确的预期效果
- ✅ 包含必要的上下文
- ✅ 避免模糊词汇（如"适当"、"合理"）
- ❌ 不要说"改进用户体验"，要说"添加进度提示"
- ❌ 不要说"优化性能"，要说"使用缓存减少重复计算"

### Step 4: 规范化表述

**表述原则**：

1. **使用祈使句**：
   - ✅ "在分析前先验证数据格式"
   - ❌ "可以考虑验证数据格式"

2. **明确主体**：
   - ✅ "Agent 应该..."
   - ❌ "应该..."

3. **量化标准**：
   - ✅ "响应时间应小于 2 秒"
   - ❌ "响应应该快速"

4. **提供理由**：
   - ✅ "使用 try-except 捕获异常，避免程序崩溃"
   - ❌ "使用 try-except"

5. **保持一致性**：
   - 与技能现有的风格和术语保持一致
   - 遵循已有的命名约定

## 问题分类标准

### 按修改位置分类

#### 1. Description 优化

**适用场景**：

- 技能触发不准确（误触发或漏触发）
- 用户对技能用途有误解
- 缺少关键的使用场景说明

**提取要点**：

- 应添加的关键词
- 应澄清的边界
- 应强调的使用时机

**示例**：

```
原始 description: "Helps with data analysis"

问题: 太宽泛，不清楚具体分析什么类型的数据

改进建议:
"Specializes in CSV and Excel data analysis, including cleaning,
transformation, and statistical summaries. Use when working with
tabular data files. Not suitable for unstructured text or images."
```

#### 2. Workflow/Instructions 补充

**适用场景**：

- Agent 遗漏关键步骤
- 执行顺序不当
- 缺少默认方案

**提取要点**：

- 缺失的步骤
- 正确的执行顺序
- 推荐的默认方法

**示例**：

```
问题: Agent 直接开始分析，未检查数据质量

改进建议:
在 Workflow 中添加：
"1. **Data Validation**: Before analysis, check for:
   - Missing values and decide handling strategy
   - Data type consistency
   - Outliers that may skew results

2. **Exploratory Analysis**: Understand the data structure..."
```

#### 3. Boundary Cases (Gotchas)

**适用场景**：

- 特殊输入导致错误
- 边界条件未处理
- 异常情况缺乏说明

**提取要点**：

- 具体的边界情况
- 正确的处理方式
- 常见的陷阱

**示例**：

```
新增 Gotchas 章节:

"## Common Pitfalls

**Empty Files**: If the input file is empty, return a clear message
instead of attempting analysis.

**Large Datasets**: For files > 100MB, recommend sampling or chunked
processing to avoid memory issues.

**Mixed Encodings**: Always detect and specify encoding when reading
files to avoid character corruption."
```

#### 4. Examples Enhancement

**适用场景**：

- 示例不足以覆盖实际场景
- 示例格式不清晰
- 缺少典型用例

**提取要点**：

- 需要补充的示例类型
- 示例的正确格式
- 示例应展示的关键点

**示例**：

````
问题: 现有示例只展示了简单查询，未展示复杂聚合

改进建议:
添加新示例:

"### Complex Aggregation Example

Input: Sales data with columns [date, product, amount, region]

Task: Monthly sales by region

Expected Output:
```json
{
  "2024-01": {
    "North": 15000,
    "South": 12000
  },
  "2024-02": {
    "North": 18000,
    "South": 13500
  }
}
```"
````

#### 5. References 组织

**适用场景**：

- 需要新增详细指南
- 现有 reference 需要更新
- 引用关系需要调整

**提取要点**：

- 新文件的内容概要
- 文件的组织结构
- SKILL.md 中的引用位置

### 按问题性质分类

| 性质           | 特征                 | 优先级 | 处理方式 |
| -------------- | -------------------- | ------ | -------- |
| **功能性缺陷** | 技能无法完成预期任务 | P0     | 立即修复 |
| **准确性问题** | 输出结果错误或不可靠 | P0-P1  | 优先修复 |
| **完整性缺失** | 遗漏重要步骤或考虑   | P1     | 尽快补充 |
| **清晰度不足** | 说明模糊导致误解     | P1-P2  | 优化表述 |
| **可用性改进** | 提升使用便利性       | P2-P3  | 酌情添加 |
| **一致性优化** | 统一风格和术语       | P3     | 批量处理 |

## 建议生成模板

### 模板 1: Description 优化建议

```markdown
## Description 优化建议

**当前描述**:
```

[复制当前的 description]

```

**问题分析**:
- [问题 1]: [具体说明]
- [问题 2]: [具体说明]

**建议修改为**:
```

[新的 description 文本]

```

**改进点**:
1. [改进点 1]: [说明为什么这样改]
2. [改进点 2]: [说明为什么这样改]

**预期效果**:
- 提高 [具体指标，如触发准确率]
- 减少 [具体问题，如误触发次数]
```

### 模板 2: 指令补充建议

```markdown
## 指令补充建议

**问题场景**:
[描述导致问题的具体场景]

**当前指令**:
```

[引用当前的相关指令，如果没有则写"无"]

````

**缺失内容**:
[详细说明缺失的步骤、说明或约束]

**建议添加**:
```markdown
[具体的指令文本，使用 markdown 格式]
````

**插入位置**:

- 文件: [SKILL.md 或 specific reference file]
- 章节: [具体章节名称]
- 位置: [在某个步骤之前/之后，或作为新章节]

**理由**:
[解释为什么需要这个指令，以及它如何解决观察到的问题]

````

### 模板 3: 边界情况文档

```markdown
## 边界情况文档建议

**边界情况**: [简要描述]

**触发条件**:
- [条件 1]
- [条件 2]

**当前行为**:
[描述技能在当前情况下的表现，通常是错误的或不理想的]

**期望行为**:
[描述应该如何正确处理]

**建议创建文件**: `references/[filename].md`

**文件内容大纲**:
```markdown
# [Boundary Case Name]

## When This Occurs
[说明什么情况下会遇到这个边界情况]

## Why It Matters
[解释为什么需要特别处理]

## How to Handle
[详细的处理步骤和方法]

## Example
[具体的示例]

## Common Mistakes
[常见的错误处理方式及为什么错误]
````

**在 SKILL.md 中的引用**:
在 [章节名] 中添加:
"- For edge cases like [this case], see [reference file](references/[filename].md)"



## 质量控制原则

### 1. 具体性原则

**要求**：每个建议都必须足够具体，可以直接执行

**检查清单**：

- [ ] 是否说明了具体要做什么？
- [ ] 是否有明确的执行步骤？
- [ ] 是否避免了模糊词汇？
- [ ] 是否提供了必要的上下文？

**反例 vs 正例**：

```
❌ "改进错误处理"
✅ "在所有文件读取操作中添加 try-except 块，
   捕获 FileNotFoundError 和 PermissionError，
   返回友好的错误消息而非堆栈跟踪"
```

### 2. 可操作性原则

**要求**：建议应该能够直接转化为代码或文档修改

**检查清单**：

- [ ] 能否直接根据建议进行修改？
- [ ] 是否需要额外的解释才能执行？
- [ ] 修改的位置是否明确？
- [ ] 修改的内容是否清晰？

### 3. 可测试性原则

**要求**：改进后的效果应该能够通过测试验证

**检查清单**：

- [ ] 能否设计测试用例验证改进？
- [ ] 是否有明确的成功标准？
- [ ] 改进前后是否有可衡量的差异？

**示例**：

```
建议: "添加输入验证"

可测试性检查:
✅ 可以构造无效输入测试验证逻辑
✅ 成功标准：无效输入返回清晰的错误消息
✅ 可衡量：验证覆盖率从 0% 提升到 X%
```

### 4. 一致性原则

**要求**：新内容与技能现有风格和内容保持一致

**检查清单**：

- [ ] 术语使用是否一致？
- [ ] 代码风格是否一致？
- [ ] 文档结构是否协调？
- [ ] 是否与现有内容冲突？

### 5. 渐进式披露原则

**要求**：保持 SKILL.md 简洁，详细内容放入 references/

**决策规则**：

- 如果内容 < 50 行且是核心流程 → 放入 SKILL.md
- 如果内容 > 50 行或是细节说明 → 放入 references/
- 如果是边界情况或高级用法 → 放入 references/
- 如果 SKILL.md 已接近 500 行 → 优先放入 references/

## 优先级排序方法

### 紧急度评估矩阵

| 维度         | 高 (3分)       | 中 (2分)     | 低 (1分)     |
| ------------ | -------------- | ------------ | ------------ |
| **影响范围** | 所有用户都遇到 | 部分用户遇到 | 少数用户遇到 |
| **严重程度** | 完全无法使用   | 显著影响体验 | 轻微不便     |
| **发生频率** | 每次都发生     | 经常发生     | 偶尔发生     |
| **修复难度** | 简单快速       | 中等工作量   | 复杂耗时     |

**优先级计算**：

```
优先级分数 = (影响范围 + 严重程度 + 发生频率) / 修复难度

分数 > 2.5: P0 (立即处理)
分数 1.5-2.5: P1 (优先处理)
分数 1.0-1.5: P2 (计划处理)
分数 < 1.0: P3 (酌情处理)
```

### 批量处理策略

**P0 问题**：

- 立即应用，不等批量
- 单独确认，快速迭代

**P1-P2 问题**：

- 按类型分组（所有 description 优化一起处理）
- 批量确认，一次性应用
- 注意相互之间的影响

**P3 问题**：

- 积累到一定数量后批量处理
- 可以与其他优化工作合并
- 避免频繁的小改动

## 常见陷阱和避免方法

### 陷阱 1: 过度泛化

**问题**：从单一案例得出过于宽泛的结论

**示例**：

```
用户纠正: "这个 CSV 文件的日期格式是 YYYY-MM-DD"

❌ 过度泛化: "所有日期都应该使用 ISO 格式"
✅ 恰当抽象: "在处理包含日期的数据时，应明确指定并验证日期格式"
```

**避免方法**：

- 寻找多个类似案例再泛化
- 使用"通常"、"一般"等限定词
- 保留一定的灵活性

### 陷阱 2: 忽略上下文

**问题**：提出的建议与技能的整体设计不符

**示例**：

```
技能定位: 轻量级快速分析工具

❌ 不当建议: "添加复杂的机器学习模型"
✅ 恰当建议: "添加基本的统计摘要功能"
```

**避免方法**：

- 理解技能的定位和目标用户
- 考虑与现有功能的协调性
- 评估实现的复杂度是否合理

### 陷阱 3: 治标不治本

**问题**：只解决表面症状，未触及根本原因

**示例**：

```
问题: Agent 多次询问同一信息

❌ 治标: "在 prompt 中强调不要重复提问"
✅ 治本: "添加信息收集的检查清单，确保一次性获取所有必要信息"
```

**避免方法**：

- 深入分析问题的根本原因
- 考虑系统性的解决方案
- 评估方案的长期有效性

### 陷阱 4: 引入新问题

**问题**：改进一个问题的同时引入了新问题

**示例**：

```
原问题: 输出太长

❌ 不当改进: "限制输出为 100 字"
  → 新问题：信息不足

✅ 恰当改进: "提供简洁版和详细版两种输出模式，默认简洁版"
```

**避免方法**：

- 评估改进的副作用
- 考虑各种使用场景
- 进行测试验证

## 总结

反馈提取的核心是将用户的零散纠正转化为系统化的改进建议：

1. **深入理解**：透过现象看本质
2. **恰当抽象**：从具体到通用的平衡
3. **具体建议**：可执行、可测试、可衡量
4. **质量控制**：遵循五大原则
5. **优先级排序**：基于影响和成本
6. **避免陷阱**：警惕常见错误

通过规范化的提取流程，确保每一条反馈都能有效提升技能质量。
