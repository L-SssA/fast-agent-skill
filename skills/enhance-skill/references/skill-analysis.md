# 技能能力分析指南

本指南提供系统化分析现有技能能力的方法，帮助识别知识缺口、评估新信息的价值，并确定增强的优先级和方向。

## 分析目标

在增强技能之前，需要深入理解：

1. **技能的当前能力**：它能做什么，不能做什么
2. **知识的覆盖范围**：已涵盖哪些主题，遗漏了哪些
3. **内容的质量**：是否清晰、完整、准确、可操作
4. **结构的合理性**：组织是否逻辑清晰、易于导航
5. **增强的机会**：哪里最需要改进，优先级如何

## 快速能力评估框架

### Step 1: 阅读 SKILL.md

SKILL.md 是技能的"名片"，提供了核心信息：

**关注点**：

- **Description**：技能的核心职责和使用时机
- **使用流程**：主要的操作步骤
- **关键能力**：声称提供的功能
- **Reference 引用**：有哪些详细文档

**提取信息**：

```markdown
技能名称: [name]
核心职责: [从 description 提取]
主要场景: [从 description 和使用流程提取]
参考文档: [列出所有引用的 reference 文件]
```

**示例**：

```markdown
技能名称: sql-optimizer
核心职责: 分析和优化 SQL 查询性能
主要场景:

- 慢查询诊断
- 索引建议
- 执行计划解读
- 查询重写
  参考文档:
- references/query-analysis.md
- references/index-strategy.md
- references/explain-plan.md
```

### Step 2: 扫描 Reference 文件

快速浏览所有 reference 文件，了解深度和广度：

**检查清单**：

- [ ] 有多少个 reference 文件？
- [ ] 每个文件的大致行数？
- [ ] 文件的组织结构（扁平 vs 嵌套）？
- [ ] 是否有明显的主题分类？
- [ ] 文件之间是否有交叉引用？

**创建文件地图**：

```markdown
references/
├── query-analysis.md (180 行) - 查询分析方法
├── index-strategy.md (220 行) - 索引策略
├── explain-plan.md (150 行) - 执行计划解读
└── common-patterns.md (90 行) - 常见优化模式
```

### Step 3: 识别核心主题

从 SKILL.md 和 reference 文件中提取核心主题：

**方法**：

1. 列出所有主要章节标题
2. 合并相似的主题
3. 形成主题列表

**示例**：

```
SQL 优化技能的核心主题：
1. 查询分析
2. 索引策略
3. 执行计划
4. 常见优化模式
5. 性能测试
```

### Step 4: 评估覆盖深度

对每个核心主题评估覆盖的深度：

**深度等级**：

- **深入**：有专门的 reference 文件，内容详细（>100 行）
- **中等**：有专门章节或文件，但内容简略（50-100 行）
- **浅层**：仅在 SKILL.md 中提及，或文件很小（<50 行）
- **缺失**：完全未涉及

**示例评估**：

```
主题覆盖深度：
1. 查询分析 - 深入（180 行专门文件）
2. 索引策略 - 深入（220 行专门文件）
3. 执行计划 - 中等（150 行，但缺少高级主题）
4. 常见优化模式 - 浅层（90 行，仅基础模式）
5. 性能测试 - 缺失（完全未涉及）
```

## 知识缺口识别

### 缺口类型

#### 1. 主题缺口 (Topic Gaps)

完全缺失的重要主题。

**识别方法**：

- 对比行业标准或最佳实践
- 查看相关技能的覆盖范围
- 考虑用户的常见问题

**示例**：

```
SQL 优化技能的知识缺口：
- 缺少"分区策略"主题
- 缺少"缓存优化"主题
- 缺少"并发控制"主题
```

#### 2. 深度缺口 (Depth Gaps)

主题存在但覆盖不够深入。

**识别方法**：

- 检查是否有高级主题
- 是否有足够的示例
- 是否覆盖了边界情况

**示例**：

```
索引策略的深度缺口：
- 只有 B-tree 索引，缺少其他索引类型
- 缺少复合索引的最佳实践
- 没有索引维护的建议
```

#### 3. 实践缺口 (Practice Gaps)

缺少实际应用的指导。

**识别方法**：

- 是否有足够的示例代码？
- 是否有真实场景的案例？
- 是否有分步操作指南？

**示例**：

```
实践缺口：
- 缺少完整的优化案例研究
- 没有性能测试的具体步骤
- 缺少工具配置指南
```

#### 4. 更新缺口 (Update Gaps)

内容过时或与最新版本不符。

**识别方法**：

- 检查版本标注
- 对比最新官方文档
- 查找 deprecated 的功能

**示例**：

```
更新缺口：
- 仍在使用 MySQL 5.7 的语法
- 缺少 MySQL 8.0 的新特性
- 某些推荐的工具已过时
```

### 缺口优先级评估

使用以下标准评估缺口的优先级：

**高优先级**（应立即补充）：

- 影响核心功能的正确使用
- 用户频繁遇到的问题
- 行业标准的必备知识
- 安全相关的缺失

**中优先级**（应计划补充）：

- 扩展技能的适用范围
- 提升技能的专业性
- 常见的进阶需求
- 性能优化的关键点

**低优先级**（可选补充）：

- 边缘场景的处理
- 高级专家级知识
- 替代方案的详细介绍
- 历史背景或理论知识

**优先级矩阵**：

|            | 高频使用 | 低频使用 |
| ---------- | -------- | -------- |
| **高影响** | 高优先级 | 中优先级 |
| **低影响** | 中优先级 | 低优先级 |

## 新信息价值评估

当接收到新的资料时，评估其对目标技能的价值：

### 相关性评分

对每条提取的信息进行相关性评分（1-5 分）：

**5 分 - 极高相关**：

- 直接支持核心功能
- 填补重要的知识缺口
- 解决常见问题

**4 分 - 高相关**：

- 扩展现有能力
- 提供改进建议
- 补充边界情况

**3 分 - 中等相关**：

- 有用但非必需
- 适用于部分场景
- 作为参考资料

**2 分 - 低相关**：

- 边缘场景
- 高级专业知识
- 与核心功能间接相关

**1 分 - 不相关**：

- 与技能职责无关
- 过于基础或过于专业
- 不应包含在此技能中

**示例评估**：

```
输入：MySQL 8.0 CTE（公用表表达式）指南

信息评估：
1. CTE 基本语法 - 4 分（高相关，扩展现有查询能力）
2. 递归 CTE 示例 - 3 分（中等，适用于特定场景）
3. CTE 性能对比 - 4 分（高相关，帮助决策何时使用）
4. CTE 历史演变 - 1 分（不相关，纯理论知识）
5. CTE 调试技巧 - 5 分（极高相关，解决常见问题）
```

### 独特性评估

评估信息是否是新的还是已有内容的重复：

**全新**：技能完全未涉及的内容
**补充**：扩展现有内容的新角度或新示例
**更新**：修正或更新过时的信息
**重复**：与现有内容基本相同

**处理策略**：

- 全新 → 高优先级整合
- 补充 → 中优先级整合
- 更新 → 根据重要性决定
- 重复 → 跳过或合并

### 可操作性评估

评估信息的可操作程度：

**高可操作性**：

- 具体的步骤和方法
- 可直接使用的代码
- 明确的配置参数

**中可操作性**：

- 概念性的指导
- 需要进一步解释
- 一般性的建议

**低可操作性**：

- 纯理论知识
- 缺乏具体指导
- 过于抽象

**优先整合高可操作性的信息**。

## 技能边界分析

### 职责边界

明确技能应该做什么和不应该做什么：

**应该在范围内**：

- Description 中明确提到的功能
- 核心职责的直接延伸
- 用户使用技能的必要知识

**应该在范围外**：

- 与其他技能重叠的功能
- 过于通用的基础知识
- 特定于某个项目的细节

**示例**：

```
sql-optimizer 技能的边界：

在范围内：
✓ SQL 查询分析和优化
✓ 索引策略和建议
✓ 执行计划解读
✓ 数据库特定的优化技巧

在范围外：
✗ 数据库安装和配置（应由 db-admin 技能负责）
✗ 应用程序架构设计（应由 architecture 技能负责）
✗ SQL 基础语法教学（代理已掌握）
✗ 特定业务逻辑的优化（太具体）
```

### 与其他技能的关系

识别相关技能和潜在的重叠：

**分析方法**：

1. 列出项目中的其他技能
2. 识别功能重叠的部分
3. 明确各自的职责边界
4. 建立协作关系

**示例**：

```
相关技能分析：

sql-optimizer:
  - 职责：查询级别的优化
  - 边界：单个查询的性能

db-admin:
  - 职责：数据库管理和配置
  - 边界：服务器级别的设置

data-modeler:
  - 职责：数据库设计和建模
  - 边界：表结构和关系设计

协作关系：
- sql-optimizer 可能建议 db-admin 调整配置
- sql-optimizer 可能建议 data-modeler 修改索引
- 三者各司其职，避免重叠
```

### 功能膨胀预警

警惕技能变得过于复杂：

**警告信号**：

- SKILL.md 接近或超过 500 行
- reference 文件超过 10 个
- 涵盖了多个不同的主题领域
- 用户反馈说技能"太复杂"或"不知道何时使用"

**解决方案**：

- 将相关但独立的功能拆分为子技能
- 将详细内容移至 references
- 重新审视和收窄 description
- 考虑创建技能家族

**示例**：

```
警告：data-analysis 技能变得过于庞大

当前状态：
- SKILL.md: 480 行
- References: 15 个文件
- 涵盖：数据清洗、可视化、统计分析、机器学习

建议拆分：
- data-cleaning: 数据清洗和预处理
- data-visualization: 数据可视化
- statistical-analysis: 统计分析
- ml-basics: 机器学习基础

每个技能职责单一，更易使用和维护。
```

## 内容质量评估

### 清晰度评估

**优秀**：

- 语言简洁明了
- 结构层次分明
- 示例充分且相关
- 术语解释清楚

**需改进**：

- 表述模糊或歧义
- 结构混乱
- 缺少示例
- 术语未解释

**检查方法**：

- 随机选择段落，是否能快速理解？
- 示例是否可以直接运行或使用？
- 是否有未定义的专业术语？

### 完整性评估

**完整**：

- 覆盖了主题的主要方面
- 包含了必要的先决条件
- 提供了充分的示例
- 说明了限制和边界

**不完整**：

- 遗漏关键步骤
- 缺少重要场景
- 示例不足
- 未说明前提条件

**检查方法**：

- 按照指南操作，能否成功完成任务？
- 是否覆盖了常见的使用场景？
- 是否有明显的"这里应该有..."的感觉？

### 准确性评估

**准确**：

- 信息与技术文档一致
- 代码示例可以正常运行
- 版本号正确
- 没有过时或废弃的内容

**不准确**：

- 与技术文档矛盾
- 代码有错误
- 版本信息错误
- 包含已废弃的方法

**检查方法**：

- 抽样验证代码示例
- 对比官方文档
- 检查版本标注
- 查找 deprecated 标记

### 可操作性评估

**高可操作性**：

- 提供了清晰的步骤
- 有完整的示例
- 说明了预期结果
- 包含故障排除

**低可操作性**：

- 只有理论描述
- 缺少具体步骤
- 示例不完整
- 没有错误处理

**检查方法**：

- 能否按照指南独立完成操作？
- 遇到问题时是否有解决指引？
- 示例是否足够参考？

## 增强机会识别

基于以上分析，识别增强的机会：

### 机会类型

#### 1. 填补缺口

补充缺失的重要主题。

**示例**：

```
机会：添加"分区策略"主题
理由：
- 高优先级缺口（影响大数据场景）
- 用户经常询问
- 有现成的技术文档可提取
行动：创建 references/partitioning-strategy.md
```

#### 2. 深化内容

扩展现有主题的深度。

**示例**：

```
机会：深化"执行计划"内容
理由：
- 当前只有基础解读
- 缺少高级优化技巧
- 有新的最佳实践可添加
行动：更新 references/explain-plan.md，添加高级主题章节
```

#### 3. 添加示例

增加实际应用的示例。

**示例**：

```
机会：添加完整的优化案例
理由：
- 当前缺少端到端案例
- 用户需要看到完整流程
- 有真实项目经验可分享
行动：创建 references/case-studies.md，包含 3 个完整案例
```

#### 4. 更新内容

修正过时或不准确的信息。

**示例**：

```
机会：更新 MySQL 版本相关内容
理由：
- 当前基于 MySQL 5.7
- MySQL 8.0 已成为主流
- 有新特性和语法变化
行动：全面审查并更新所有 MySQL 相关内容
```

#### 5. 改进结构

优化内容的组织和导航。

**示例**：

```
机会：重组参考文档结构
理由：
- 当前文件过多且分散
- 用户难以找到需要的信息
- 可以按照使用流程重新组织
行动：合并相关文件，创建清晰的导航结构
```

### 机会优先级排序

使用以下标准对增强机会排序：

**影响力**（1-5 分）：

- 对用户有多大帮助？
- 是否解决常见问题？
- 是否显著提升技能价值？

**工作量**（1-5 分，分数越高工作量越大）：

- 需要多少时间完成？
- 是否需要大量研究？
- 是否需要重构现有内容？

**优先级计算**：

```
优先级 = 影响力 / 工作量
```

**优先级分类**：

- **高优先级**：高影响力 + 低工作量（快速获胜）
- **中优先级**：高影响力 + 高工作量 或 中影响力 + 中工作量
- **低优先级**：低影响力 或 工作量过大

**示例排序**：

```
增强机会优先级：

1. 添加 CTE 调试技巧（优先级：高）
   - 影响力：5（解决常见问题）
   - 工作量：2（已有资料，容易整合）
   - 优先级：5/2 = 2.5

2. 创建完整案例研究（优先级：中）
   - 影响力：5（极大帮助用户）
   - 工作量：4（需要整理多个案例）
   - 优先级：5/4 = 1.25

3. 更新 MySQL 版本内容（优先级：中）
   - 影响力：4（影响所有用户）
   - 工作量：5（需要全面审查）
   - 优先级：4/5 = 0.8

4. 重组文档结构（优先级：低）
   - 影响力：3（改善体验但非必需）
   - 工作量：5（重大重构）
   - 优先级：3/5 = 0.6
```

## 分析工作流程

### Step 1: 准备阶段

收集必要的信息：

- 目标技能的完整目录结构
- SKILL.md 和所有 reference 文件
- 用户提供的待整合资料
- 相关的行业标准或最佳实践

### Step 2: 快速评估

执行快速能力评估框架：

- 阅读 SKILL.md
- 扫描 reference 文件
- 识别核心主题
- 评估覆盖深度

**时间**：5-10 分钟

### Step 3: 缺口分析

系统性地识别知识缺口：

- 列出所有主题
- 评估每个主题的覆盖深度
- 识别缺失的主题
- 评估缺口的优先级

**时间**：10-15 分钟

### Step 4: 新信息评估

评估待整合信息的价值：

- 对每条信息进行相关性评分
- 评估独特性（是否重复）
- 评估可操作性
- 筛选出高价值的信息

**时间**：10-20 分钟（取决于信息量）

### Step 5: 边界检查

确认增强不会破坏技能边界：

- 检查是否在职责范围内
- 识别与其他技能的重叠
- 预警功能膨胀风险
- 必要时提出拆分建议

**时间**：5-10 分钟

### Step 6: 质量评估

评估现有内容的质量：

- 清晰度
- 完整性
- 准确性
- 可操作性

识别需要改进的地方。

**时间**：10-15 分钟

### Step 7: 机会识别

基于以上分析，识别增强机会：

- 列出所有可能的增强
- 评估影响力和工作量
- 计算优先级
- 排序得到增强路线图

**时间**：10-15 分钟

### Step 8: 制定增强方案

综合所有分析结果，制定具体的增强方案：

- 确定要添加的内容
- 确定整合方式
- 预估工作量
- 向用户呈现方案并获得确认

**时间**：5-10 分钟

**总时间**：约 60-100 分钟（对于复杂技能）

## 总结

技能能力分析是增强工作的基础。通过系统化的分析，可以：

1. **全面了解**：深入理解技能的当前状态
2. **精准定位**：识别最有价值的增强机会
3. **合理优先级**：基于影响力和工作量排序
4. **保持边界**：避免功能膨胀和职责混乱
5. **质量保证**：确保增强后的技能质量更高

记住核心要点：

- 先分析后行动，不要盲目增强
- 使用结构化框架，确保全面性
- 量化评估，基于数据做决策
- 保持技能边界，避免过度扩展
- 持续迭代，根据反馈调整

通过本指南的方法，可以系统化地分析和增强任何技能，使其更加完善和实用。
