# 整合策略指南

本指南提供将提取的信息智能整合到现有技能包中的系统化方法，确保增强后的技能保持清晰的结构、良好的可维护性和一致的quality。

## 核心原则

### 1. 最小侵入性

优先选择对现有结构影响最小的整合方式：

- 能追加就不修改
- 能创建新文件就不改动现有文件
- 能放在 references 就不放在 SKILL.md

### 2. 渐进式披露维护

始终遵循渐进式披露原则：

- SKILL.md 保持在 500 行以内
- 详细内容放入 references/ 目录
- 在 SKILL.md 中提供清晰的引用指引

### 3. 职责单一性

确保增强不会破坏技能的单一职责：

- 不引入与核心功能无关的内容
- 避免技能变得过于复杂
- 必要时建议拆分为多个技能

### 4. 一致性保持

与现有内容的风格和结构保持一致：

- 使用相同的标题层级
- 遵循已有的命名约定
- 保持相似的详细程度
- 统一的格式和标记

## 整合决策树

使用以下决策流程确定最佳的整合方式：

```
开始
  ↓
信息是否与现有内容重复？
  ↓ 是 → 跳过或合并到现有内容
  ↓ 否
信息量大小？
  ↓ 小（< 50 行）→ 可以添加到现有文件
  ↓ 大（≥ 50 行）→ 考虑创建新文件
  ↓
信息类型？
  ↓ 新增知识 → 创建新 reference 文件
  ↓ 最佳实践 → 更新现有 practice 文件或创建新的
  ↓ 边界情况 → 添加到 Gotchas 章节或创建 edge-cases.md
  ↓ 示例代码 → 添加到 examples 章节或创建示例文件
  ↓ 注意事项 → 添加到 Gotchas 或 Warnings 章节
  ↓
是否影响 SKILL.md 的描述或指令？
  ↓ 是 → 评估是否需要更新 SKILL.md
  ↓ 否 → 仅更新 references
  ↓
更新后 SKILL.md 是否超过 500 行？
  ↓ 是 → 将部分内容移至 references
  ↓ 否 → 完成
```

## SKILL.md 更新策略

### 何时更新 SKILL.md

仅在以下情况下更新 SKILL.md：

✓ 需要添加对新 reference 文件的引用  
✓ description 需要调整以反映新增能力  
✓ 需要添加重要的使用步骤或注意事项  
✓ 工作流程发生重大变化  
✗ 仅添加详细内容（应放在 references 中）  
✗ 添加示例代码（应放在 references 中）  
✗ 添加技术细节（应放在 references 中）

### 更新 Description

当增强显著扩展了技能的能力范围时，可能需要更新 description：

**原则**：

- 保持长度 < 1024 字符
- 仍然包含"做什么"和"何时使用"
- 添加关键的新触发词
- 不要过度详细

**示例**：

```yaml
# 更新前
description: 帮助分析和优化 SQL 查询性能。提供索引建议、查询重写指导和执行计划解读。

# 更新后（添加了新的能力）
description: 帮助分析和优化 SQL 查询性能。提供索引建议、查询重写指导、执行计划解读和数据库配置优化。支持 MySQL、PostgreSQL 和 SQLite。
```

### 添加 Reference 引用

当创建新的 reference 文件时，必须在 SKILL.md 中添加引用：

**位置**：在"何时加载详细指南"章节

**格式**：

```markdown
- **新功能说明** → 读取 [`references/new-feature.md`](references/new-feature.md)
  - 简要说明这个文件包含什么内容
```

**示例**：

```markdown
## 何时加载详细指南

- **基础查询优化** → 读取 [`references/query-optimization.md`](references/query-optimization.md)
  - 包含索引策略、JOIN 优化、子查询改写等

- **高级性能调优** → 读取 [`references/advanced-tuning.md`](references/advanced-tuning.md) [NEW]
  - 包含分区策略、缓存优化、并发控制等高级主题
```

### 更新使用流程

如果增强引入了新的使用步骤，更新"使用流程"章节：

**原则**：

- 保持步骤数量在 4-8 步之间
- 每个步骤简洁明了
- 突出新增的步骤

**示例**：

```markdown
## 使用流程

1. **连接数据库**: 建立数据库连接
2. **分析查询**: 使用 EXPLAIN 分析执行计划
3. **识别瓶颈**: 定位性能问题所在
4. **应用优化**: 根据建议进行优化 [NEW]
5. **验证改进**: 重新运行测试确认效果 [NEW]
6. **记录变更**: 保存优化前后的对比
```

## Reference 文件组织原则

### 文件命名规范

- 使用小写字母、数字、连字符
- 名称应清晰反映内容
- 避免过于通用的名称（如 `info.md`）
- 与目录中其他文件保持一致的命名风格

**好的示例**：

- `query-optimization.md`
- `edge-cases.md`
- `best-practices.md`
- `api-reference.md`

**不好的示例**：

- `info.md` （太模糊）
- `New_Features.md` （使用了大写和下划线）
- `stuff.md` （无意义）

### 文件结构设计

每个 reference 文件应有清晰的结构：

```markdown
# 文件标题

简短的介绍段落，说明这个文件的内容和用途。

## 主要章节 1

详细内容...

### 子章节（如需要）

更多细节...

## 主要章节 2

更多内容...

## 示例（如适用）

提供实际示例...

## 参考资料（如适用）

外部链接和进一步阅读...
```

### 文件粒度控制

**单个文件应该**：

- 聚焦一个主题或相关的一组主题
- 长度在 50-300 行之间
- 可以独立理解和引用

**避免**：

- 单个文件超过 500 行（考虑拆分）
- 文件少于 20 行（考虑合并）
- 一个文件涵盖多个不相关的主题

### 文件间关系管理

当多个 reference 文件之间存在关联时：

1. **相互引用**：在相关章节添加交叉引用

   ```markdown
   有关更多详细信息，参见 [高级配置](advanced-config.md)。
   ```

2. **避免循环依赖**：保持引用方向清晰

3. **在 SKILL.md 中说明关系**：

   ```markdown
   - **基础配置** → 读取 [`references/basic-config.md`](references/basic-config.md)
     - 包含必需的配置项和默认值

   - **高级配置** → 读取 [`references/advanced-config.md`](references/advanced-config.md)
     - 包含性能调优、安全设置等高级选项
     - 建议在掌握基础配置后再阅读
   ```

## 内容去重和冲突解决

### 检测重复内容

在整合前检查：

1. **完全重复**：相同的信息已存在
   - 处理：跳过或更新现有内容

2. **部分重叠**：新信息与现有内容有交集
   - 处理：合并相关内容，保留更全面版本

3. **互补信息**：新信息补充现有内容
   - 处理：添加到适当位置

4. **冲突信息**：新信息与现有内容矛盾
   - 处理：需要用户决策，标注版本或上下文差异

### 去重策略

**策略 1：合并相似内容**

```markdown
# 更新前（两个文件都有超时配置）

# file1.md

- 连接超时：30 秒

# file2.md

- 读取超时：60 秒

# 更新后（合并到一个文件）

# timeout-config.md

- 连接超时：30 秒
- 读取超时：60 秒
- 写入超时：60 秒
```

**策略 2：引用而非复制**

```markdown
# 不要在多个文件中重复相同内容

# 而是在一个地方定义，其他地方引用

详见 [超时配置](timeout-config.md#connection-timeout)。
```

**策略 3：版本化差异**

```markdown
# 如果是因为版本不同导致的差异

## v1.x 配置

- 超时设置：30 秒

## v2.x 配置（推荐）

- 超时设置：60 秒
- 新增重试机制
```

### 冲突解决流程

当发现冲突时：

1. **识别冲突类型**
   - 事实性冲突（哪个是正确的？）
   - 偏好性冲突（哪种方法更好？）
   - 上下文冲突（适用于不同场景？）

2. **收集信息**
   - 来源的可靠性
   - 适用的版本或环境
   - 社区共识或官方推荐

3. **向用户呈现选项**

   ```markdown
   检测到关于超时配置的冲突：

   选项 A（来自官方文档 v2.0）：

   - 超时：60 秒

   选项 B（来自现有技能）：

   - 超时：30 秒

   建议：采用选项 A，因为它是最新官方推荐。
   您希望如何处理？
   ```

4. **记录决策**
   - 在文件中添加注释说明选择的原因
   - 保留被替代内容的引用（如有价值）

## 渐进式披露维护

### SKILL.md 行数控制

**目标**：保持 SKILL.md < 500 行

**策略**：

1. **定期审查**：每次增强后检查行数
2. **内容迁移**：当接近限制时，将详细内容移至 references
3. **摘要替代**：用简短摘要替代长篇内容，并提供链接

**示例**：

```markdown
# 迁移前（在 SKILL.md 中）

## 配置详解

这里有很长的配置说明...（100 行）

# 迁移后

## 配置详解

详细的配置说明请参考 [配置指南](references/configuration.md)。

快速开始：

1. 复制默认配置文件
2. 修改必要的参数
3. 运行验证脚本

完整说明见 [配置指南](references/configuration.md#detailed-explanation)。
```

### References 深度控制

**避免过深的嵌套**：

- ✓ `references/basic.md`
- ✓ `references/advanced/topics.md`
- ✗ `references/a/b/c/d.md`

**建议**：最多两层嵌套，优先扁平结构

### 引用清晰度

确保所有引用都清晰明确：

```markdown
# 好的引用

详见 [性能优化指南](references/performance.md#database-tuning)。

# 不好的引用

详见那里。（哪里？）
```

## 整合模式库

### 模式 1：新增独立主题

**场景**：提取的信息形成一个独立的主题

**操作**：

1. 创建新的 reference 文件
2. 在 SKILL.md 的"何时加载详细指南"中添加引用
3. 如需要，在使用流程中添加相关步骤

**示例**：

```
新增：缓存策略
→ 创建 references/caching-strategy.md
→ 在 SKILL.md 中添加：
  - **缓存优化** → 读取 [`references/caching-strategy.md`](references/caching-strategy.md)
```

### 模式 2：扩展现有主题

**场景**：提取的信息补充现有的 reference 文件

**操作**：

1. 找到最相关的现有文件
2. 在适当位置添加新内容
3. 保持原有的结构和风格
4. 如文件过大，考虑拆分

**示例**：

```
新增：更多的错误处理示例
→ 更新 references/error-handling.md
→ 在"常见错误"章节添加新示例
→ 检查文件行数，如超过 300 行考虑拆分
```

### 模式 3：添加 Gotchas

**场景**：提取了常见的陷阱和注意事项

**操作**：

1. 检查是否有现有的 Gotchas 章节
2. 如有，添加到该章节
3. 如无，创建新的 gotchas.md 或在相关文件中添加
4. 在 SKILL.md 中强调这个重要章节

**示例**：

````markdown
## 常见陷阱

### 陷阱 1：忘记关闭资源

**问题**：打开的文件或连接未正确关闭导致泄漏

**解决**：

```python
with open('file.txt') as f:
    content = f.read()
# 自动关闭
```
````

**详情**：参见 [资源管理](references/resource-management.md)。

```

### 模式 4：添加示例库

**场景**：提取了多个有用的代码示例

**操作**：
1. 创建 examples/ 子目录（如不存在）
2. 为每个示例创建独立文件或使用分类文件
3. 在 reference 文件中引用这些示例
4. 确保示例有清晰的说明和上下文

**示例**：
```

examples/
├── basic-usage.py
├── advanced-pattern.py
└── error-handling.py

在 references/usage.md 中：

## 示例代码

基础用法参见 [basic-usage.py](../examples/basic-usage.py)。
高级模式参见 [advanced-pattern.py](../examples/advanced-pattern.py)。

````

### 模式 5：重组内容结构

**场景**：增强后发现现有结构不够合理

**操作**：
1. 设计新的组织结构
2. 向用户说明重组的理由和好处
3. 获得确认后执行重组
4. 更新所有相关的引用
5. 保留变更记录

**注意**：重组是重大变更，需谨慎执行并充分沟通。

## 版本控制和变更记录

### 变更记录格式

在每个 enhanced 的技能中添加变更记录：

```markdown
## 变更记录

### v1.1 (2026-04-17)
- 新增：缓存策略指南
- 更新：添加了 5 个常见陷阱
- 改进：重构了配置说明，提高清晰度

### v1.0 (2026-03-01)
- 初始版本
````

### 变更类型标识

- **新增**：添加了全新的内容
- **更新**：修改了现有内容
- **改进**：优化了结构或表达
- **修复**：纠正了错误或过时信息
- **移除**：删除了过时或不相关的内容

### 向后兼容性

尽量保持向后兼容：

- 不删除现有的 reference 文件（如必须，先标记为 deprecated）
- 不改变文件的 URL 路径
- 如必须 breaking change，在变更记录中明确说明

## 整合工作流程

### Step 1: 分析提取的信息

回顾从 extraction-guide 中提取的信息：

- 按类别分组
- 评估每条信息的重要性
- 识别与现有内容的关系

### Step 2: 检查现有结构

分析目标技能的当前结构：

- 有哪些 reference 文件
- 每个文件的主要内容
- SKILL.md 的当前行数
- 现有的组织逻辑

### Step 3: 制定整合方案

为每条信息确定整合方式：

- 创建新文件还是更新现有文件
- 添加到哪个章节
- 是否需要更新 SKILL.md
- 是否会触发行数限制

### Step 4: 向用户展示方案

清晰地呈报整合方案：

```markdown
## 整合方案

基于提取的信息，建议进行以下增强：

### 1. 创建新文件

- `references/caching-strategy.md` (约 120 行)
  - 包含 3 种缓存策略
  - 性能对比数据
  - 实施指南

### 2. 更新现有文件

- `references/performance.md`
  - 添加缓存相关章节（约 40 行）
  - 更新性能优化清单

### 3. 更新 SKILL.md

- 在"何时加载详细指南"中添加缓存策略引用
- 预计增加 10 行，当前 320 行 → 更新后 330 行

### 4. 添加示例

- `examples/caching-example.py`
  - 完整的缓存实现示例

总计：新增 ~170 行内容

是否执行此方案？或有其他偏好？
```

### Step 5: 执行整合

获得用户确认后：

1. 按方案创建/更新文件
2. 保持一致的格式和风格
3. 添加必要的交叉引用
4. 更新 SKILL.md 中的引用

### Step 6: 验证整合

检查整合质量：

- [ ] 所有新文件都有正确的 frontmatter（如需要）
- [ ] 引用链接都有效
- [ ] 没有 broken links
- [ ] SKILL.md 行数在限制内
- [ ] 内容与现有风格一致
- [ ] 没有重复内容
- [ ] 变更记录已更新

### Step 7: 生成增强报告

使用 enhancement-template.md 生成报告：

- 总结所做的变更
- 列出新增的内容
- 说明整合的理由
- 提供后续建议

## 常见错误和避免方法

### 错误 1：过度更新 SKILL.md

**问题**：将太多详细内容放入 SKILL.md，导致超过 500 行

**避免**：严格遵循渐进式披露，详细内容放入 references

### 错误 2：创建过多小文件

**问题**：每个小点都创建独立文件，导致文件碎片化

**避免**：相关的小主题合并到一个文件中，保持合理的文件粒度

### 错误 3：忽略引用更新

**问题**：创建了新的 reference 文件但忘记在 SKILL.md 中添加引用

**避免**：使用检查清单，确保所有新文件都被正确引用

### 错误 4：破坏现有结构

**问题**：重组时改变了文件路径或名称，导致外部链接失效

**避免**：尽量避免 breaking changes，如必须进行，充分沟通并提供迁移指南

### 错误 5：不一致的风格

**问题**：新增内容的格式、语气与现有内容不一致

**避免**：仔细研究现有内容的风格，保持一致性
